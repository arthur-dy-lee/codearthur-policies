# App 提审前检查清单

适用对象：首次上架 iOS App（App Store Connect + App Review）  
使用方式：提交审核前逐项打勾，任何一项不确定都先补齐再提审。

---

## 0. 发布策略（先做对，再做快）

- [ ] 本次只提审 1 个主力 App（首发建议）
- [ ] 预留 7-14 天处理驳回与补件
- [ ] 明确版本目标：本次仅做“可过审最小版本”（MVP），不在临门一脚加功能

---

## 1. 账号与权限准备

- [x] Apple Developer Program 账号状态正常（个人/公司）
- [x] App Store Connect 里有该 App 记录（Bundle ID 一致）
- [ ] 证书、Profiles、Signing 配置正确，Archive 可成功
- [x] 个人开发者：无需团队分工角色；若后续有协作者，再补充 App Manager/Developer 权限
- [ ] 若是免费且无内购：可跳过 Paid Apps 相关检查；若有内购/付费项目：仍需完成 Paid Apps Agreement、税务与银行信息

---

## 2. App 基础合规（最常见驳回点）

- [ ] App 可打开，不闪退，不黑屏，不卡死
- [ ] 关键路径可用：启动 -> 核心功能 -> 退出
- [ ] 没有“占位文案”“开发测试开关”“Demo 假数据”暴露给审核员
- [ ] 若需登录：提供可用审核账号（含密码/验证码方案/测试环境说明）
- [ ] 若是订阅或付费：价格、时长、权益描述一致且清晰
- [x] 没有误导性功能宣称（写了 AI/去广告/高清导出就必须可用）

---

## 3. 隐私与权限声明一致性

- [ ] `Info.plist` 中权限用途说明完整且具体（相机、相册、麦克风、定位等）
- [x] App 内实际请求的权限，与权限说明文字、隐私政策一致
- [ ] App Store Connect 的“App Privacy”问卷已填写且与实际一致
- [ ] 若接入第三方 SDK（广告/统计/归因），已核对其数据收集行为
- [x] 隐私政策 URL 可公开访问，非空、非 404、内容可读
- [x] 如有账号系统，提供账号删除入口（如适用）
- [x] 本地文档（如 `privacy.html`、`terms.html`、`account-deletion.html`）已部署到公网可访问地址并填入 App Store Connect

---

## 4. 订阅与内购（如适用）

- [ ] 所有内购商品在 App Store Connect 已创建并可提交
- [ ] Paywall 展示：价格、计费周期、自动续费说明、恢复购买入口
- [x] “服务条款（Terms）”和“隐私政策（Privacy）”链接可点击
- [x] App 内有“恢复购买”并可触达
- [x] 不用“限时误导/虚假原价”诱导订阅
- [ ] 沙盒环境测试通过（新购、续订、取消、恢复）

---

## 5. 内容与法律风险

- [x] 不包含侵权素材（音乐、视频、图片、字体、商标）
- [x] 不包含色情、仇恨、暴力极端、违法内容
- [ ] 若含 UGC（用户内容），有举报/屏蔽/审核机制（如适用）
- [x] AI 生成内容有基本安全策略（敏感内容拦截或说明）
- [x] 医疗/金融/法律建议类功能有免责说明与资质边界（如适用）

---

## 6. App Store 物料完整性

- [x] App 名称、副标题、关键词不违规（不蹭词、不堆砌品牌）
- [x] 描述真实，不夸大，不承诺不可控结果
- [ ] 截图与预览视频展示的是“真实功能”和当前版本界面
- [x] 各本地化文案无明显机翻错误、错别字、乱码
- [ ] 分类、年龄分级、内容声明准确
- [x] Support URL、Marketing URL、Privacy URL 都可访问

---

## 7. 技术质量与稳定性

- [ ] Release 包在真机测试通过（至少 2 台不同机型）
- [ ] 弱网/断网/首次启动/权限拒绝场景可正常处理
- [ ] 无明显卡顿、死循环、持续高耗电
- [ ] Crash 监控已接入（如 Firebase Crashlytics）
- [ ] 关键错误有用户可理解提示，不是技术报错堆栈
- [x] 版本号/构建号规范递增（CFBundleShortVersionString / Build）

---

## 8. 审核备注（Review Notes）准备

- [ ] 用英文写清楚 App 核心用途（2-4 句）
- [x] 如需登录，提供测试账号、密码、步骤
- [ ] 如有特殊路径（如必须先开权限/导入样本），写成 1-2-3 步骤
- [x] 如某功能受地区/时间限制，明确说明
- [x] 如接入 AI/外部服务，说明失败兜底逻辑（避免审核误判为不可用）

可复制模板：

```text
App purpose:
This app helps users [核心价值一句话].

Review account:
Email: [xxx]
Password: [xxx]
If verification is needed, use: [方法]

How to test core feature:
1) Open app and tap [入口]
2) Grant [permission] if prompted
3) Use sample content at [位置] to verify [功能]

Notes:
- Subscription is optional and can be restored via [Restore Purchases].
- Privacy Policy: [URL]
```

---

## 9. 提审前 30 分钟“最后一遍”

- [ ] 清理测试开关、调试按钮、临时弹窗
- [ ] 再次核对权限弹窗文案与实际行为一致
- [ ] 再次核对隐私政策/服务条款 URL 可访问
- [ ] 再次确认截图与当前版本 UI 一致
- [ ] 审核备注已填完整、账号可登录
- [ ] 归档包来自最新 commit，且可复现

---

## 10. 被驳回后的处理 SOP（建议）

- [ ] 先读清 Guideline 编号，逐条拆成“问题-证据-修复”
- [ ] 24 小时内提交第一次回复（态度清晰、证据具体）
- [ ] 修复后重新上传并在 Review Notes 指明改动点
- [ ] 若误判，礼貌申诉并附截图/录屏/测试账号
- [ ] 连续 2 次同类驳回时，暂停加新功能，先专注过审

---

## 建议节奏（首发）

1. 第 1 周：完成上述清单并提审第一个 App  
2. 第 2 周：处理反馈、拿到首个通过经验  
3. 第 3 周：复用流程提审第二个 App（可并行准备第三个）

> 原则：先把“通过率”跑出来，再把“上架速度”提上去。
